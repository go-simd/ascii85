# ascii85

[![CI](https://github.com/go-simd/ascii85/actions/workflows/ci.yml/badge.svg)](https://github.com/go-simd/ascii85/actions/workflows/ci.yml)
![coverage](https://img.shields.io/badge/coverage-100%25-brightgreen)

A drop-in fast path for the Adobe/btoa **ascii85** (base-85) encoding,
byte-, error-, and offset-identical to the standard library's
[`encoding/ascii85`](https://pkg.go.dev/encoding/ascii85). The bulk of **both
directions** (the regular 4-byte â†” 5-char path) runs a SIMD kernel generated by
[go-asmgen](https://github.com/go-asmgen/asmgen); everything the kernel must not
guess about â€” encode's all-zero `z` shortcut and short trailing fragment,
decode's whitespace, `z`, short/invalid groups, flush, and `CorruptInputError`
offsets â€” reuses the standard library, so the output, errors, and offsets match
`encoding/ascii85.Encode`/`Decode` exactly.

```go
import "github.com/go-simd/ascii85"

n := ascii85.Encode(dst, src)               // identical bytes to encoding/ascii85.Encode
m := ascii85.MaxEncodedLen(len(src))        // sizing
nd, ns, err := ascii85.Decode(dst, src, true) // identical (ndst, nsrc, err), incl. CorruptInputError offsets
```

The API mirrors the standard library 1:1 â€” `Encode(dst, src) int`,
`Decode(dst, src, flush) (ndst, nsrc int, err error)`, `MaxEncodedLen(n) int`,
and a re-exported `CorruptInputError` type â€” so it is a literal drop-in.

## How the SIMD encode works

The hard part of base-85 is the repeated **division by 85**. The kernel avoids
real division with a reciprocal multiply that is **exact over the entire 32-bit
range** (verified exhaustively over all 2ÂłÂ˛ values):

```
v / 85  ==  (v * 0xC0C0C0C1) >> 32 >> 6        remainder = v - (v/85)*85
```

Five such steps yield the five base-85 digits of each 4-byte group (most
significant first), `'!'` (33) is added, and a fixed cross-lane shuffle scatters
the per-lane digits into the stride-5 output layout (16 input bytes â†’ 20 output
chars per 128-bit block). The wrapper hands the kernel only maximal runs of
groups that contain **no all-zero group**, so every kernel group expands to
exactly 5 chars; an all-zero group (which the spec shortens to `z`) and the
trailing fragment go to `encoding/ascii85`, keeping the output byte-identical.

## How the SIMD decode works

Decode is the inverse of encode and needs **no division at all** â€” each 5-char
group is a base-85 **multiply-accumulate**:

```
v = (((c0*85 + c1)*85 + c2)*85 + c3)*85 + c4      (each ck = char - '!')
```

evaluated per 32-bit lane with multiply-low (`PMULLD`/`VMULF`/â€¦) and add (the low
32 bits are exact for any valid encoding). A fixed cross-lane gather pulls the
five stride-1 chars of each group into the lanes (20 input chars â†’ 16 output
bytes per 128-bit block), and the decoded value is written big-endian. The
wrapper hands the kernel only maximal leading runs of **clean** 5-char groups â€”
groups whose every byte is a regular base-85 digit `'!'`..`'u'` (33..117) â€” and
caps the run by the available `dst` room; the first byte that is whitespace, the
`z` shortcut, or an out-of-range/short/invalid group ends the run and the
**remainder is handed to `encoding/ascii85.Decode`**, with any `CorruptInputError`
offset shifted back to the absolute position. So whitespace skipping, `z`, the
short trailing group, `flush`, the `(ndst, nsrc)` returns, and the error offsets
all stay identical to the standard library.

| op | amd64 | ppc64le | s390x | riscv64 | loong64 | arm64 |
|---|---|---|---|---|---|---|
| encode | **SSE** (PMULULQ) | **VSX** (VMULEUW/OUW) | **vector** (VMLHF) | **RVV** (VMULHUVV) | **LSX** (VMUHWU) | fused scalar |
| decode | **SSE** (PMULLD) | **VSX** (VMULUWM) | **vector** (VMLF) | **RVV** (VMULVX) | **LSX** (VMULW) | fused scalar |

**Five of the six 64-bit targets ship a real SIMD kernel in both directions.**
The one exception is **arm64**: Go's released arm64 assembler exposes **no vector
integer multiply** (`VMUL`/`VUMULL`/`VMLS` exist only in an unreleased upstream
`cmd/asm` patch), and neither base-85's encode `/85` nor the decode
multiply-accumulate can be done with the shift/mask tricks that the sibling
[`go-simd/base64`](https://github.com/go-simd/base64) uses for its NEON kernel.
arm64 therefore uses a **fused scalar** path in both directions (no per-group
`z`/whitespace branch, slice-bounded loads/stores) which is still measurably
faster than `encoding/ascii85` (measured natively on Apple Silicon:
~1.9Ă— encode, ~1.8Ă— decode).

### Per-arch encode-kernel notes

The 32-bit mulhi is the only primitive that differs across backends; the digit
math, the `+33`, and the stride-4â†’stride-5 scatter are otherwise identical.

- **amd64 (SSE):** `PMULULQ` does the even/odd 32Ă—32â†’64 products, blended back
  with `PAND`/`PANDN`; `PMULLD` does `q*85`; `PSHUFB` scatters (a stride-4 pack
  via `PSLLO` byte shifts, expanded to stride 5 + the 5th digit). A per-lane
  `PSHUFB` byte-reverse builds the big-endian group value on load.
- **ppc64le (VSX):** `LXVB16X` byte-order-correct load (its in-register words are
  big-endian, so the group value needs **no** byte reversal); mulhi via
  `VMULEUW`/`VMULOUW` (high32 at words 0,2) merged with `VMRGEW`, `>>6` via
  `VSRW`; `VMULUWM` for `q*85`; `VPERM` scatter (gaps via an index into an
  all-zero operand, since ppc `VPERM` has no `PSHUFB`-style 0x80 zeroing); two
  overlapping `STXVB16X` windows write the 20 bytes.
- **s390x (vector facility, big-endian):** `VL` (group value is natively
  big-endian, no reversal); single-op `VMLHF` mulhi, `VESRLF $6`; `VMLF`/`VSF`
  for the remainder; `VSLDB` byte shifts; `VPERM` scatter (gap index â†’ zero
  operand); `VST` + `VSTEF` store.
- **riscv64 (RVV):** vl pinned to 4 (SEW=32) for arithmetic and 16 (SEW=8) for
  byte loads/shuffles/stores via `VSETIVLI` (needs VLENâ‰Ą128); single-op
  `VMULHUVV` mulhi, `VSRLVI $6`; `VMULVV` for `q*85`; `VRGATHERVV` for the
  per-lane byte reverse **and** the scatter (out-of-range index â†’ 0 gives free
  gap zeroing).
- **loong64 (LSX):** single-op `VMUHWU` mulhi, `VSRLW $6`; `VMULW` for `q*85`;
  `VSHUFB` for the byte-reverse and the scatter (its first operand is the control
  vector; index â‰Ą16 selects the zero high source â†’ gap zeroing).

### Per-arch decode-kernel notes

Decode needs only a multiply-low + add MAC, so it is simpler than encode. The
gather (window byte `5l+k` â†’ low byte of lane `l`, gaps zeroed) uses each
backend's byte shuffle with one shared control; the multiply-low and the output
big-endian layout are the only other differences.

- **amd64 (SSE):** five `MOVOU src+k` + `PSHUFB` gather C0..C4, `PSUBL` the `'!'`,
  the MAC is `PMULLD`/`PADDL`, and a per-lane `PSHUFB` byte-reverse precedes one
  16-byte `MOVOU` store.
- **ppc64le (VSX):** five `LXVB16X src+k` + `VPERM` gather (gap index 16 â†’ all-zero
  operand); `VSUBUWM` for `-'!'`, MAC via `VMULUWM`/`VADDUWM`. Big-endian: the
  in-register word is stored big-endian by `STXVB16X`, so **no byte reversal**.
- **s390x (vector facility, big-endian):** five `VL src+k` + `VPERM` gather into the
  low byte of each fullword; `VSF` for `-'!'`, MAC via `VMLF`/`VAF`. Big-endian:
  the fullword lane is already in output order, so a plain `VST` stores 16 bytes
  with **no reversal** (verified position-dependently under QEMU).
- **riscv64 (RVV):** vl pinned 16 (SEW=8) for the `VRGATHERVV` gathers / store and
  4 (SEW=32) for arithmetic; `VSUBVX`/`VMULVX`/`VADDVV` for the MAC; a `VRGATHERVV`
  byte-reverse before one `VSE8V` store (out-of-range gather index â†’ 0 = free gap
  zeroing).
- **loong64 (LSX):** five `VMOVQ src+k` + `VSHUFB` gather (index 16 â†’ zero high
  source); `VSUBW`/`VMULW`/`VADDW` MAC; a `VSHUFB` byte-reverse before one `VMOVQ`
  store.

Every backend's instruction operand orders and shuffle/lane semantics were
**confirmed empirically under QEMU** before the full kernel was assembled (e.g.
s390x `VSF` is second-minus-first and `VPERM` indexes byte addresses with no
0x80 zeroing; ppc `VSRW Vdata,Vshift`; loong64 `VSHUFB Vctrl,Vlow,Vhigh,Vd`;
RVV `VRGATHERVV Vindex,Vsrc,Vd`). All six targets are verified against
`encoding/ascii85` (table + `FuzzEncode`/`FuzzDecode`, byte- and error-identical):
amd64 natively (x86_64 VM), arm64 natively, and ppc64le/s390x/riscv64/loong64
under QEMU.

## Performance

- **arm64** (Apple Silicon, native, 1 MiB buffer): the fused-scalar paths run at
  **~1920 MB/s vs ~1010 MB/s** stdlib on encode (**~1.9Ă—**) and
  **~2060 MB/s vs ~1130 MB/s** stdlib on decode (**~1.8Ă—**), purely from dropping
  the per-group `z`/whitespace branch and using slice-bounded loads/stores.
- **amd64**: the SSE kernels are correctness-validated (table + fuzz) on an
  x86_64 VM. Honest caveat: the only available x86_64 host is itself QEMU-TCG
  (no native silicon was available), and TCG makes SIMD disproportionately
  expensive relative to scalar, so its throughput is not representative of native
  hardware. Native amd64 throughput is **pending real silicon**.
- **ppc64le / s390x / riscv64 / loong64**: real SIMD kernels, **QEMU-validated
  (byte-, error-, and offset-identical to `encoding/ascii85`); native perf
  pending** real silicon â€” no GitHub-hosted POWER/IBM Z/RISC-V/LoongArch runners
  exist and QEMU TCG is not cycle-accurate.

## Coverage

The CI gate enforces **100% coverage of the Go code** on every arch job: native
amd64 + native arm64, plus QEMU jobs for ppc64le, s390x, riscv64, and loong64.
Coverage is of the Go statements only â€” the generated `.s` SIMD kernels are not
measured by `go test -cover`; they are validated by differential tests against
the `encoding/ascii85` reference (byte-, error-, and offset-identical) plus
`FuzzEncode`/`FuzzDecode` on every arch that ships a kernel.

## Regenerating the kernels

The `.s` files are committed; the generators (`*_gen.go`, tagged
`//go:build ignore`) reproduce them with go-asmgen:

```sh
export GOWORK=off
go get github.com/go-asmgen/asmgen@v0.5.0
go run encode_gen.go            # amd64 encode
go run encode_ppc64x_gen.go     # ppc64le encode
go run encode_s390x_gen.go      # s390x encode
go run encode_riscv64_gen.go    # riscv64 encode
go run encode_loong64_gen.go    # loong64 encode
go run decode_gen.go            # amd64 decode
go run decode_ppc64x_gen.go     # ppc64le decode
go run decode_s390x_gen.go      # s390x decode
go run decode_riscv64_gen.go    # riscv64 decode
go run decode_loong64_gen.go    # loong64 decode
go mod edit -droprequire github.com/go-asmgen/asmgen
go mod edit -go=1.21
go mod tidy
```
